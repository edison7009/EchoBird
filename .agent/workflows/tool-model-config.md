# Tool Model Config Standard

All Echobird standalone tools (Reversi, BudgetCoder, etc.) share the **same model config source**.
This document is the single source of truth for how tools read and write model configuration.

---

## 1. Canonical Model Store Path

```
~/.echobird/config/models.json
```

| Platform | Actual path |
|---|---|
| Windows | `C:\Users\{user}\.echobird\config\models.json` |
| macOS | `/Users/{user}/.echobird/config/models.json` |
| Linux | `/home/{user}/.echobird/config/models.json` |

This file is written and managed by the Echobird desktop app.
Tools should **read-only** unless they are explicitly model management tools.

---

## 2. models.json Schema

The file is a **JSON array** of model objects:

```json
[
  {
    "internalId": "m-1aa838",
    "name": "MiniMax-M2.7",
    "modelId": "MiniMax-M2.7",
    "baseUrl": "https://api.minimaxi.com/v1",
    "apiKey": "sk-...",
    "anthropicUrl": "https://api.minimaxi.com/anthropic",
    "type": "CLOUD"
  }
]
```

### Field Reference

| Field | Type | Required | Description |
|---|---|---|---|
| `internalId` | string | ✅ | Unique ID generated by Echobird (e.g. `m-1aa838`) |
| `name` | string | ✅ | Display name shown in UI |
| `modelId` | string | ✅ | Model identifier sent in API request (e.g. `gpt-4o`) |
| `baseUrl` | string | ✅ | OpenAI-compatible endpoint base URL |
| `apiKey` | string | ✅ | API key (may be empty for local models) |
| `anthropicUrl` | string | ❌ | Anthropic-protocol endpoint (if model supports it) |
| `type` | string | ❌ | `"CLOUD"` or `"LOCAL"` |

> [!NOTE]
> `modelId` is what goes in the `model` field of the API request body.
> `name` is for display only — do NOT send it to the API.

## 3. Tool Config File Pattern

Each tool declares its own config file at:
```
tools/{tool-name}/config.json
```

This file tells the tool WHERE to find model config and HOW to map fields:

```json
{
    "docs": "https://...",
    "configFile": "~/.echobird/config/models.json",
    "format": "json",
    "read": {
        "modelId": ["modelId"],
        "baseUrl": ["baseUrl"],
        "apiKey":  ["apiKey"]
    },
    "write": {
        "modelId":  "modelId",
        "baseUrl":  "baseUrl",
        "apiKey":   "apiKey"
    }
}
```

### Real example — Reversi (`tools/reversi/config.json`)

```json
{
    "configFile": "~/.echobird/reversi.json",
    "read": {
        "model":   ["model"],
        "baseUrl": ["openai.baseUrl", "baseUrl"],
        "apiKey":  ["openai.apiKey",  "apiKey"]
    },
    "write": {
        "model":             "model",
        "openai.baseUrl":    "baseUrl",
        "openai.apiKey":     "apiKey",
        "anthropic.baseUrl": "baseUrl",
        "anthropic.apiKey":  "apiKey"
    }
}
```

## 4. Standalone Mode (without Echobird)

When a tool runs **without Echobird installed** (e.g. BudgetCoder on another machine):

**Fallback lookup order:**
1. `~/.echobird/config/models.json` — shared Echobird store (preferred)
2. `~/.{tool-name}/models.json` — tool-specific standalone store (fallback)
3. Environment variable: `ECHOBIRD_MODELS_PATH` — override for CI/scripts

**Standalone models.json** (same schema, placed at `~/.{tool-name}/models.json`):
```json
[
  {
    "internalId": "m-local-01",
    "name": "GPT-4o",
    "modelId": "gpt-4o",
    "baseUrl": "https://api.openai.com/v1",
    "apiKey": "sk-..."
  }
]
```

> [!TIP]
> For BudgetCoder: if `~/.echobird/config/models.json` exists, prefer it automatically.
> Only fall back to `~/.budgetcoder/models.json` if Echobird is not installed.

## 5. API Call Format

All tools use the **OpenAI-compatible** format:

```http
POST {baseUrl}/chat/completions
Authorization: Bearer {apiKey}
Content-Type: application/json

{
  "model": "{modelId}",
  "messages": [...]
}
```

For **Anthropic protocol** (when `anthropicUrl` is set):
```http
POST {anthropicUrl}/messages
x-api-key: {apiKey}
anthropic-version: 2023-06-01
```

---

## 6. Adding a New Tool

When building a new standalone tool:

1. Create `tools/{tool-name}/config.json` with the field mapping
2. Read from `~/.echobird/config/models.json` (prefer) or `~/.{tool-name}/models.json` (fallback)
3. Use `internalId` as the unique key for persistence (e.g. "last used model")
4. Never hardcode model names — always load from the config file
5. Never write back to `~/.echobird/config/models.json` (read-only from tool perspective)

---

## 7. Encryption Note

The `apiKey` field in `models.json` is stored **in plaintext** (no encryption at the file level).
The Echobird app encrypts keys **in memory** during transport but the file itself is plain JSON.
Tools can read the key directly without any decryption step.
