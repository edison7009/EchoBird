# 26. 通用 CLI Bridge 技术路线

> **状态：开发中 · 优先级：P1 · 分支：`feature/cli-bridge`**  
> 目标：让 EchoBird 的 Bridge 能连接任意 AI CLI 工具，并为 Agent 提供角色身份选择。

---

## 一、功能概述

Bridge CLI 分为三个核心部分：

### 1. 多 CLI 通信

通过统一的通信层连接多种 Agent CLI 工具（本地 + SSH 远程），包括：

| CLI | 厂商 | 协议 | 状态 |
|---|---|---|---|
| OpenClaw | EchoBird | JSON stream over PTY | ✅ 已支持 |
| Claude Code | Anthropic | `claude --print` one-shot | 待实现 |
| OpenCode | OpenCode.ai | headless one-shot | 待调研 |
| Qwen Code | Alibaba | TBD | 待调研 |
| 通用 CLI | — | 原始文本 I/O | 兜底方案 |

- 已适配的 CLI 工具：自动协议匹配、会话管理、参数构建
- 未适配的开源 CLI：通过通用 PTY 模式兜底支持
- 远程连接：通过 SSH 检测服务器上已安装的 Agent 进程/目录

### 2. Agent 角色身份系统

连接 Agent 后，用户可为其分配专业角色身份。基于 [agency-agents](https://github.com/msitarzewski/agency-agents)（MIT 协议）提供的 120+ 专业角色 Prompt：

**选择模式：**

- 顶部分类 Tab 筛选（All / Engineering / Design / Marketing / Data & AI / Product / Testing）
- 6 列竖版卡片网格（5:8 竖版比例），图片居中裁切
- 悬停滑入：默认只显示角色名，悬停展开分割线 + 描述
- 点击即选（✓ 角标），无需二次确认
- 选中后 Tab 栏显示角色名替换工具名

**数据存储：** 角色 .md 文件打包到 `roles/` 目录，随 Bridge CLI 一起部署，无需远程下载。

### 3. 角色交易平台与加密保护

> **核心认知：选择角色 > 安装技能。** 角色 = 雇佣一个熟练工（自带技能和经验），比让 AI 从零学习调用技能更高效。优质角色 Prompt 是核心知识产权，值得付费。

**商业模式：**

| 类型 | 来源 | 加密 | 价格 |
|---|---|---|---|
| 默认角色集 | 随 Echobird 内置 | ❌ 明文 | 免费 |
| 社区角色 | 用户自建 + 分享 | ❌ 明文 | 免费 |
| 付费角色 | 官方 / 第三方创作者 | ✅ 加密 | 按角色定价 |
| 订阅角色包 | 官方精选集 | ✅ 加密 | 月/年订阅 |

**加密架构（参考 [EasyClaw 防破解分析](easyclaw-anti-tamper-analysis.md) 纵深防御思路）：**

```
roles/engineering/premium-architect/
├── cover.jpg              ← 明文（展示需要）
├── meta.json              ← 明文（name, desc, price, license_type）
└── role.enc               ← 加密的 System Prompt（替代明文 role.md）
```

**三层防护设计：**

```
层1: 内容加密（核心）
     role.enc = AES-256-GCM(role.md, derived_key)
     derived_key = HKDF(master_key, role_id + machine_id)
     → 密文绑定角色 + 机器，复制到其他机器无法解密

层2: 运行时解密（防磁盘泄露）
     Bridge CLI 进程内解密 → 直接注入 Agent → 内存释放
     明文 System Prompt 永远不落盘
     → 参考 EasyClaw: "配置真相由受控进程持有"

层3: 许可证验证（防盗版）
     启动时验证 license_key → 服务端校验 → 返回解密密钥
     离线缓存有时间窗口（如 7 天 grace period）
     → 参考 EasyClaw: runtime migration 每次启动校验
```

**角色文件格式对比：**

| | 免费角色 | 付费角色 |
|---|---|---|
| Prompt 文件 | `role.md`（明文） | `role.enc`（加密） |
| 元数据 | `role.md` frontmatter | `meta.json`（明文，用于展示） |
| 封面 | `cover.jpg` | `cover.jpg` |
| 前端展示 | 直接读取 | 读取 `meta.json`（无需解密） |
| 注入 Agent | 直接读取内容 | Bridge CLI 运行时解密后注入 |

**平台功能规划：**

- 角色商店页面（浏览、搜索、分类筛选）
- 创作者后台（上传角色 → 自动加密 → 上架）
- 购买/激活流程（支付 → 下发 license_key → 下载 role.enc）
- 评分评价体系（角色效果评估）
- 收入分成（创作者 70% / 平台 30%）

**架构预留要点（当前需要做的）：**

1. `roles/` 目录结构**同时支持 `role.md`（明文）和 `role.enc`（加密）**，扫描时自动识别
2. `meta.json` 预留 `license_type` 字段（`free` / `paid` / `subscription`）
3. Bridge CLI 预留 `decrypt_role()` 函数接口，当前对明文文件直接 passthrough
4. 角色选择器 UI 预留付费标识（如 💎 角标）

## 二、架构设计

```
前端 Channels 页面
    ↓  tauri invoke: bridge_start / bridge_chat / bridge_stop
Bridge Manager (Rust, channel_commands.rs)
    ↓  根据 plugin.json 的 protocol 字段动态分派
    ├── stdio-json     → 现有 Bridge Binary 模式 (OpenClaw)
    ├── cli-oneshot    → 每次聊天 spawn CLI 子进程 (Claude Code, OpenCode)
    └── cli-interactive → 持久 PTY 子进程 (通用 CLI, Phase 3)
```

### 协议类型

| protocol | 描述 | 生命周期 |
|---|---|---|
| `stdio-json` | 通过 bridge binary (stdin/stdout JSON) 持久通信 | 持久进程 |
| `cli-oneshot` | 每次聊天 spawn CLI 子进程执行一次，读 stdout | 一次性 |
| `cli-interactive` | 持久 PTY 子进程 + ready_signal 检测 | 持久进程 |

## 三、Plugin 配置格式

每个 CLI 工具用一个 `plugin.json` 描述，存放于 `plugins/{id}/`：

```json
{
    "id": "claudecode",
    "name": "Claude Code",
    "protocol": "cli-oneshot",
    "cli": {
        "command": "claude",
        "detectCommand": "claude --version",
        "args": ["--print", "--output-format", "text"],
        "resumeArgs": ["--print", "--continue", "{sessionId}"],
        "sessionArg": "--session-id",
        "sessionMode": "always",
        "systemPromptArg": "--system-prompt",
        "systemPromptWhen": "new-session"
    }
}
```

| 字段 | 说明 |
|---|---|
| `protocol` | `stdio-json` / `cli-oneshot` / `cli-interactive` |
| `systemPromptArg` | CLI 注入 System Prompt 的参数名 |
| `systemPromptWhen` | `new-session`（仅首次）/ `always`（每次） |

## 四、Agent 角色数据结构

### 设计原则

- **约定大于配置**：固定目录结构，app 扫描即用
- **自包含**：每个角色一个文件夹，包含图片 + 角色文件
- **开放可扩展**：用户可自行添加/复制角色目录
- **我们提供默认集**，用户自定义与我们无关

### 目录布局

```
roles/                           ← 开放目录，用户可自行添加
├── engineering/                 ← 分类目录
│   ├── frontend-developer/      ← 角色目录（= 角色 ID）
│   │   ├── cover.jpg            ← 封面图片（固定文件名）
│   │   └── role.md              ← 角色定义（名称 + 描述 + System Prompt）
│   ├── backend-architect/
│   │   ├── cover.jpg
│   │   └── role.md
│   └── ...
├── design/
│   └── ...
├── marketing/
│   └── ...
└── custom/                      ← 用户自定义分类
    └── my-role/
        ├── cover.jpg
        └── role.md
```

### 角色文件格式 (`role.md`)

```markdown
---
name: Frontend Developer
desc: Modern web UI specialist
---
You are a senior frontend developer specializing in...
(System Prompt 正文)
```

- `name` — 显示名称（支持任何语言，用户自行填写）
- `desc` — 简短描述（1-2 行）
- 正文 — System Prompt，发送给 Agent

### 运行时扫描

App 启动或打开角色选择器时：
1. 扫描 `roles/` 下所有 `分类/角色/role.md`
2. 解析 frontmatter 获取 `name` + `desc`
3. 检测 `cover.jpg` 是否存在
4. 按分类目录名分组显示

## 六、角色注入流程

> 核心问题：远程 Bridge CLI 包含了 `roles/` 目录，用户在 Echobird UI 选角色后，怎么把文件放到正确位置？

**答：Echobird 只传 role_id，Bridge CLI 自己处理文件分发。**

```
Echobird UI (本地)                    Bridge CLI (远程服务器)
   │                                      │
   │  选择角色                              │
   │  ────── role_id ──────────────────►   │
   │         "engineering/frontend-dev"    │
   │                                      ├── 读取 roles/engineering/frontend-dev/role.md
   │                                      │
   │                                      ├── 根据当前 Agent 类型分发：
   │                                      │   ├── Claude Code → cp role.md → 项目/CLAUDE.md
   │                                      │   ├── OpenCode   → cp role.md → 项目/AGENTS.md
   │                                      │   └── 通用       → 作为 --system-prompt 参数传入
   │                                      │
   │  ◄──── 确认 "role applied" ──────────  │
```

### Bridge CLI 角色命令

| 命令 | 说明 |
|---|---|
| `bridge_set_role(role_id)` | 读取 role.md → 写入 Agent 对应位置 |
| `bridge_get_roles()` | 扫描 roles/ 目录 → 返回可用角色列表（含 name/desc/cover） |
| `bridge_clear_role()` | 移除当前角色身份文件 |

### Echobird 侧角色列表来源

两个方案（可并行）：

1. **远程扫描**：Echobird 调用 `bridge_get_roles()` 获取远程 roles/ 目录内容
   - 优点：数据源唯一，确保一致性
   - 缺点：首次加载有延迟，需要 Bridge CLI 先连接
2. **本地缓存**：首次连接时拉取角色列表缓存到本地
   - 优点：秒开角色选择器
   - 缺点：可能不同步

建议：**首次远程扫描 + 本地缓存**，后续打开选择器用缓存，有刷新按钮手动同步。

## 七、角色身份持久化策略

不同 Agent 的记忆机制差异大，需分别对待：

| Agent | 注入方式 | 持久化 | 风险 | 策略 |
|---|---|---|---|---|
| Claude Code | 写入 `CLAUDE.md` | `.claude/` 记忆目录 | 低 | 写文件，每轮自动重载 |
| OpenCode | `AGENTS.md` 或 instructions | 无内置记忆 | 高 | 每次会话头部注入 |
| OpenClaw | 同 OpenCode | 无内置记忆 | 高 | 每次会话头部注入 |
| Qwen Code | 待研究 | 待研究 | 待确认 | 按通用方案兜底 |

**通用兜底：** Echobird 侧每次发消息时，自动在用户输入前追加简短身份 Reminder。
