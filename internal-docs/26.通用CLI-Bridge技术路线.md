# 26. 通用 CLI Bridge 技术路线

> **状态：开发中 · 优先级：P1 · 分支：`feature/cli-bridge`**  
> 目标：让 EchoBird 的 Bridge 能连接任意 AI CLI 工具，并为 Agent 提供角色身份选择。

---

## 一、功能概述

Bridge CLI 分为两个核心部分：

### 1. 多 CLI 通信

通过统一的通信层连接多种 Agent CLI 工具（本地 + SSH 远程），包括：

| CLI | 厂商 | 协议 | 状态 |
|---|---|---|---|
| OpenClaw | EchoBird | JSON stream over PTY | ✅ 已支持 |
| Claude Code | Anthropic | `claude --print` one-shot | 待实现 |
| OpenCode | OpenCode.ai | headless one-shot | 待调研 |
| Zeroclaw | 第三方 | CLI JSON | 待调研 |
| 通用 CLI | — | 原始文本 I/O | 兜底方案 |

- 已适配的 CLI 工具：自动协议匹配、会话管理、参数构建
- 未适配的开源 CLI：通过通用 PTY 模式兜底支持
- 远程连接：通过 SSH 检测服务器上已安装的 Agent 进程/目录

### 2. Agent 角色身份系统

连接 Agent 后，用户可为其分配专业角色身份。基于 [agency-agents](https://github.com/msitarzewski/agency-agents)（MIT 协议）提供的 120+ 专业角色 Prompt：

**两层选择结构：**

```
第一层: Division（大分类）
   Engineering / Design / Marketing / Game Dev / Sales / Testing / ...
   + 自定义分类（情感陪伴、虚拟人物等）

第二层: Role（具体角色）
   如 Engineering → Frontend Developer / Backend Architect / DevOps / ...
```

选中角色后，对应的 System Prompt（.md 文件）将注入 CLI 工具的启动参数。

**数据存储：** agency-agents 的角色 .md 文件打包到 `plugins/agents/` 目录，随 Bridge CLI 一起部署，无需远程下载。

## 二、架构设计

```
前端 Channels 页面
    ↓  tauri invoke: bridge_start / bridge_chat / bridge_stop
Bridge Manager (Rust, channel_commands.rs)
    ↓  根据 plugin.json 的 protocol 字段动态分派
    ├── stdio-json     → 现有 Bridge Binary 模式 (OpenClaw)
    ├── cli-oneshot    → 每次聊天 spawn CLI 子进程 (Claude Code, OpenCode)
    └── cli-interactive → 持久 PTY 子进程 (通用 CLI, Phase 3)
```

### 协议类型

| protocol | 描述 | 生命周期 |
|---|---|---|
| `stdio-json` | 通过 bridge binary (stdin/stdout JSON) 持久通信 | 持久进程 |
| `cli-oneshot` | 每次聊天 spawn CLI 子进程执行一次，读 stdout | 一次性 |
| `cli-interactive` | 持久 PTY 子进程 + ready_signal 检测 | 持久进程 |

## 三、Plugin 配置格式

每个 CLI 工具用一个 `plugin.json` 描述，存放于 `plugins/{id}/`：

```json
{
    "id": "claudecode",
    "name": "Claude Code",
    "protocol": "cli-oneshot",
    "cli": {
        "command": "claude",
        "detectCommand": "claude --version",
        "args": ["--print", "--output-format", "text"],
        "resumeArgs": ["--print", "--continue", "{sessionId}"],
        "sessionArg": "--session-id",
        "sessionMode": "always",
        "systemPromptArg": "--system-prompt",
        "systemPromptWhen": "new-session"
    }
}
```

| 字段 | 说明 |
|---|---|
| `protocol` | `stdio-json` / `cli-oneshot` / `cli-interactive` |
| `systemPromptArg` | CLI 注入 System Prompt 的参数名 |
| `systemPromptWhen` | `new-session`（仅首次）/ `always`（每次） |

## 四、Agent 角色数据结构

### 目录布局

```
plugins/agents/
├── index.json           ← 角色索引（Division + Role 列表）
├── engineering/
│   ├── frontend-developer.md
│   ├── backend-architect.md
│   └── ...
├── design/
│   ├── ui-designer.md
│   └── ...
├── game-development/
│   ├── game-designer.md
│   └── ...
└── custom/              ← 自定义分类（情感陪伴、虚拟人物等）
    └── ...
```

### 索引格式 (`index.json`)

```json
{
    "divisions": [
        {
            "id": "engineering",
            "icon": "💻",
            "name": { "en": "Engineering", "zh": "工程开发" },
            "roles": [
                { "id": "frontend-developer", "name": { "en": "Frontend Developer", "zh": "前端开发" } },
                { "id": "backend-architect", "name": { "en": "Backend Architect", "zh": "后端架构" } }
            ]
        }
    ]
}
```

角色 System Prompt 文件路径规则：`plugins/agents/{division}/{role-id}.md`

   - **角色选择器**：两层弹窗（Division → Role）
