# Universal CLI Bridge Technical Roadmap

## Architecture Overview

```
EchoBird Main App (bundles bridge/ binaries for ALL platforms)
    │
    ├── Local Channel
    │   └── Spawns bridge-{platform} subprocess ↔ stdin/stdout JSON protocol ↔ Agent CLI
    │
    └── Remote Channel
        └── SSH → pipes JSON to ~/echobird/echobird-bridge on remote → Agent CLI
```

## Directory Structure

```
D:\Echobird\
├── bridge/                    ← Central bridge binary directory (ALL platforms)
│   ├── bridge-win.exe         ← Windows x86_64 (440 KB)
│   ├── bridge-linux-x86_64    ← Linux x86_64 (1.94 MB)
│   ├── bridge-linux-aarch64   ← Linux ARM64 (1.67 MB)
│   ├── bridge-darwin-x86_64   ← macOS Intel (1.72 MB)
│   ├── bridge-darwin-aarch64  ← macOS Apple Silicon (1.51 MB)
│   ├── src/                   ← Bridge Rust source code (NOT bundled in release)
│   ├── Cargo.toml             ← Bridge build config (NOT bundled in release)
│   └── target/                ← Bridge build output (NOT bundled in release)
│
├── plugins/                   ← Agent plugin configs (one per Agent tool)
│   ├── openclaw/plugin.json   ← OpenClaw CLI args, protocol, detection
│   ├── claudecode/plugin.json ← Claude Code CLI args
│   ├── opencode/plugin.json   ← OpenCode CLI args
│   └── zeroclaw/plugin.json   ← ZeroClaw CLI args
│
└── src-tauri/
    └── tauri.conf.json        ← Bundles bridge/ and plugins/ in release
```

## Key Design Decisions

### 1. One Bridge, All Platforms
- Bridge binary is the SAME codebase compiled for 5 platform targets
- Local and remote channels use IDENTICAL JSON protocol
- Only the transport differs: local = subprocess stdin/stdout, remote = SSH pipe

### 2. Path Resolution Priority
`get_bridge_path()` in `plugin_manager.rs`:
1. **First**: Check `bridge/` directory with arch-specific name (e.g. `bridge-linux-x86_64`)
2. **Fallback**: Check `plugins/{agent_id}/` with plugin.json-defined name (legacy)

### 3. Bridge ≠ Plugin
- `bridge/` = communication binaries (5 platform builds, shared by all agents)
- `plugins/` = agent-specific configs (CLI args, protocol type, detection commands)
- Bridge handles: `chat`, `set_role`, `detect_agents`, `start_agent`
- Plugin config tells Bridge HOW to invoke each agent's CLI

## JSON Protocol (stdio-json)

All messages are single-line JSON on stdin, responses on stdout:

```json
// Set Role (system prompt injection)
→ {"type": "set_role", "agent_id": "openclaw", "role_id": "ai-engineer", "url": "https://echobird.ai/docs/roles/en/ai-engineer.md"}
← {"type": "role_set"}

// Chat
→ {"type": "chat", "message": "Hello", "session_id": "abc-123"}
← {"type": "response", "text": "Hi! How can I help?", "session_id": "abc-123"}
← {"type": "done"}

// Detect Agents
→ {"type": "detect_agents"}
← {"type": "agents", "agents": [{"id": "openclaw", "name": "OpenClaw", "installed": true, "running": true}]}
```

## Deployment Modes

### Local Channel
1. EchoBird spawns `bridge/{platform-binary}` as subprocess
2. Communicates via stdin/stdout JSON
3. Bridge invokes agent CLI (e.g. `openclaw agent --json --agent main --message "..."`)
4. Role injection: `set_role` → Bridge downloads role MD → injects as system prompt

### Remote Channel
1. EchoBird establishes SSH connection to remote server
2. Pipes JSON through SSH to `~/echobird/echobird-bridge` on remote
3. Same protocol as local — Bridge handles agent CLI invocation on remote
4. Role injection: same `set_role` protocol

### Offline Remote Deployment
When remote server has no internet access:
1. Mother Agent detects remote OS/arch via `uname -s -m`
2. Maps to correct binary from local `bridge/` directory
3. SCPs binary to `~/echobird/echobird-bridge` on remote via `upload_file` tool
4. Sets executable permissions
5. Starts Bridge

## Build & CI

### tauri.conf.json Resources
```json
"resources": [
    "../tools/",
    "../plugins/",
    "../bridge/",
    "../roles/*.json"
]
```

### CI Bridge Compilation
- GitHub Actions compiles bridge from `bridge/src/` for all 5 targets
- Compiled binaries placed in `bridge/` alongside source
- Tauri bundler packages bridge binaries into installer
- Source code (`src/`, `target/`, `Cargo.*`) should NOT be in production builds

### Important Notes
- Bridge binaries are versioned with EchoBird releases
- If bridge protocol changes, ALL 5 binaries must be recompiled
- Bridge source code lives in `bridge/` but is compiled separately from main app
- CI should clean `bridge/src/` and `bridge/target/` before Tauri build
